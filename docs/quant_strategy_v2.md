# Quant 交易策略（更全面版本）

## 1. 目标与边界

本策略面向“基于多周期震荡结构关键点（X/Y）与布林带（上/中/下轨）”的计划式交易：

- 数据层：从 Webman 订阅获取实时价格、各周期布林带、各周期震荡结构关键点（X=下方关键位、Y=上方关键位）。
- 分析层：实时识别“突破上下轨 + 关键点簇”形成的交易机会，产出交易计划（Plan）。
- 执行层：按计划在目标区间挂单/追单/止损止盈，并管理仓位与风险。

本策略不依赖主观 K 线形态；所有信号基于价格、关键点、布林带与简单统计量（距离、宽度、节奏）。

---

## 2. 核心数据结构（建议落地为内存状态）

### 2.1 周期集合与分层

- 长期：4h、1h
- 中期：30m、15m
- 短期：5m、1m

策略中所有判断都以“周期层级”为组织单位，避免只看单一周期导致频繁反向。

### 2.2 每周期状态（TimeframeState）

对每个周期 interval 维护：

- `bands`: `{ up, mb, dn }`（最新一根可未收盘 K 的布林带值）
- `bandWidth`: `up - dn`
- `pricePos`: 当前价格相对中轨的位置：`(price - mb) / bandWidth`（归一化，便于跨周期比较）
- `breakout`: 最近一次突破事件（无/上破/下破），以及事件发生时刻与突破幅度：
  - 下破幅度：`(dn - low) / bandWidth`
  - 上破幅度：`(high - up) / bandWidth`
- `keypointsX`: 最近活跃 X 关键点价格队列（建议保留 N=6~12）
- `keypointsY`: 最近活跃 Y 关键点价格队列（建议保留 N=6~12）

### 2.3 全局状态（GlobalState）

- 当前价格 `P`
- 各周期 `TimeframeState` 映射
- 活跃交易计划列表 `plans[]`
- 冷却时间/重复触发保护（同一周期同向 breakout 在短时间内只允许一次计划）

## 3. 信号总纲：先“破轨”，再“找点”，最后“定区间”

你原文的核心原则可以总结为三段式：

1) 必须先出现“价格破上下轨之外”的事件（对应某个周期）。  
2) 破轨之后，去寻找多个周期的关键点（X 或 Y）来构建“合理开单区间”。  
3) 目标与计划周期要匹配：短期以 1m/5m 为主，中期 15m/30m，长期 1h/4h。

策略落地时建议按下面的“优先级与过滤条件”组织。

---

## 4. 趋势背景（Regime）识别：决定“顺势做突破”还是“逆势做均值回归”

### 4.1 多周期位置一致性

计算长期与中期的背景方向：

- `BG_DOWN`: 4h、1h、30m、15m 的 `pricePos < 0` 占比 >= 3/4
- `BG_UP`: 4h、1h、30m、15m 的 `pricePos > 0` 占比 >= 3/4
- 其它：震荡/过渡态

### 4.2 策略风格选择

当出现破轨事件时，根据背景选择“计划风格”：

- 背景 `BG_DOWN`：
  - 短期：允许下破后做多（均值回归到中轨/上轨），但必须更严格的风控与更快的止盈。
  - 中长期：更倾向于“反弹做空”或等待更大周期上破失败信号。
- 背景 `BG_UP`：对称处理。

这一步的意义：把你的“4h/1h/30m/15m 都在中轨下方 -> 先做多目标在小周期下轨外面，再逐级扩大”的想法，变成可计算的背景开关，避免每次都拍脑袋。

---

## 5. 破轨事件（Breakout Event）定义与过滤

对每个周期 interval，定义破轨：

- 下破：`low < dn`
- 上破：`high > up`

建议增加过滤条件（减少噪音）：

- 突破幅度阈值：`breakoutStrength >= sMin`（例如 0.05~0.15，按 bandWidth 归一化）
- 触发节流：同周期同方向在 `cooldownMs` 内只触发一次
- 关联周期要求：
  - 短期计划：必须由 1m 或 5m 触发破轨
  - 中期计划：必须由 15m 或 30m 触发破轨
  - 长期计划：必须由 1h 或 4h 触发破轨

---

## 6. 关键点使用规则：用“点簇”替代“单点”

### 6.1 多周期关键点池

破轨后，构建候选关键点集合（按方向）：

- 做多（来自下破）：收集各周期 `X`（价格下方关键点）
- 做空（来自上破）：收集各周期 `Y`（价格上方关键点）

建议按“计划周期层级”限制来源：

- 短期计划：1m、5m（可加 15m 作为兜底）
- 中期计划：15m、30m（可加 1h 作为兜底）
- 长期计划：1h、4h（可加更大周期或仅用布林带）

### 6.2 XY 关键点编号：X1..Xn / Y1..Yn

为了让策略能表达“在第几号关键点做长期计划/预判点位”，建议对每个 `oscillation_structures` 的关键点做统一编号：

- X 关键点（下方支撑侧）：`X1, X2, X3, ...`
- Y 关键点（上方压力侧）：`Y1, Y2, Y3, ...`

编号建议按“与当前价格的距离（同方向）从近到远”排序后编号：

- 做多侧（使用 X）：把所有候选 X 点按 `abs(P - price)` 从小到大排序，最近的为 `X1`，其次为 `X2`，依次类推
- 做空侧（使用 Y）：同理，最近的为 `Y1`，其次为 `Y2`

用这种编号方式，你可以稳定表达：

- `X1/Y1`：更贴近当前价格、通常更容易先被触达，用于制定“挂单开仓区间”
- `X2/Y2`：比 `X1/Y1` 更远、更“结构化”的二级目标/关键位，可用于“长期持仓计划的主假设点位”（即预判点）

“确认/未确认”的处理建议：

- `X1/Y1` 更偏“已在结构里出现且更近”的执行锚
- `X2/Y2` 更偏“结构推演出的预判点位”，可能尚未在最新结构中被确认，但可作为长期计划的核心假设

### 6.3 关键点簇（Cluster）计算

将候选关键点按价格聚类（不需要复杂算法，简单桶即可）：

- 桶宽 `bucket = max( bandWidth(shortTf) * 0.05, P * 0.001 )`
- 把关键点按 `floor(price / bucket)` 分组
- 每组得到：
  - `clusterPrice`：组内均值或中位数
  - `clusterWeight`：组内点数 + 周期权重（周期越大权重越高）

为什么要簇：单个关键点很容易被“误差 + 滑点 + 噪音”穿透；簇更稳定，也更接近你说的“通过这些关键点计算出合理区间”。

## 7. 开仓目标区间（Entry Zone）计算：给执行层一个“可挂单的区间”

### 7.1 做多（下破后）

输入：

- `P` 当前价
- 触发周期 `tfTrigger` 的 `dn`（下轨）
- 关键点簇（来自 X）

构建开仓区间的推荐规则（可按优先级逐条尝试，命中即用）：

1) 若存在“最靠近 P 的 X 簇”且其距离小于 `dMax`：
   - `anchor = nearestClusterPrice`
   - `zone = [anchor - buf, anchor + buf]`
2) 若关键点呈“逐步抬高”（你原文说的“每个关键点都比前一个更高”）且 `dn` 已低于最近 X：
   - 认为短周期支撑失效，升级到更大周期：
   - `anchor = mb(15m) 或 mb(30m)`（取距离 P 最近者）
   - `zone = [anchor - buf2, anchor + buf2]`
3) 兜底：以触发周期下轨为锚：
   - `anchor = dn(tfTrigger)`
   - `zone = [anchor - buf, anchor + buf]`

缓冲 `buf` 建议取：

- `buf = max( bandWidth(tfTrigger) * 0.03, P * 0.0008 )`
- 越短周期 buf 越小，越长周期 buf 越大

### 7.2 做空（上破后）

完全对称：

- 关键点使用 `Y` 簇
- 兜底锚为 `up(tfTrigger)`
- 升级周期时使用 `mb(15m/30m)` 作为“回落做空锚”

---

## 8. 目标价（TP）与止损（SL）：与周期绑定的“分层退出”

### 8.1 目标价分层（示例）

做多（下破后均值回归）建议至少两级目标：

- `TP1`：触发周期 `mb(tfTrigger)` 附近（快速止盈）
- `TP2`：更大周期 `mb(nextTf)` 或 `dn(nextTf)` 外侧（中期目标）

你原文的“4h/1h/30m/15m 都在中轨下方 -> 先做多目标在 15m 下轨外面，再是 30m、1h、4h 下轨外面”可以整理为：

- 背景越偏空，越优先把第一目标放在更近处（例如 1m/5m 的 mb），第二目标才逐级扩展到更大周期的轨道位。

做空对称处理。

### 8.2 止损

止损必须可计算，且跟计划周期绑定：

- 短期计划：SL 放在触发周期轨道外再加缓冲：
  - 做多：`SL = dn(tfTrigger) - k * bandWidth(tfTrigger)`
  - 做空：`SL = up(tfTrigger) + k * bandWidth(tfTrigger)`
  - `k` 可取 0.05~0.15
- 中长期计划：SL 可参考更大周期的关键点簇（被有效突破则止损）

### 8.3 时间止损（Time Stop）

均值回归策略非常依赖“回归速度”，建议增加时间止损：

- 短期：超过 `N` 根 1m/5m 仍未回到 `mb(tfTrigger)`，降低仓位或平仓

---

## 8.4 长期计划（4h/1h）的 Xn/Y n（n>1）规则：用更深一档点位做主假设，用前序点位做执行参考

长期持仓计划的核心约束更新为：

- 计划周期只允许使用 `1h` 与 `4h`
- 长期计划必须绑定 `Xn/Yn` 且 `n>1`（至少 `X2/Y2`，后续 `X3/X4...`、`Y3/Y4...` 也都允许）
- `Xn/Yn` 属于主假设点位（thesisRef），允许是预判点位（可能尚未确认）
- 实际开仓区间不直接挂在 `Xn/Yn` 上，而是基于“前序点位集合”构建（entryRefs）：
  - 预判 `X2`：参考 `X1`
  - 预判 `X3`：参考 `X1, X2`
  - 预判 `X4`：参考 `X1, X2, X3`
  - 做空对称：预判 `Y3` 参考 `Y1, Y2` 等

### 8.4.1 点位来源与跨周期引用

- `1h` 计划：只使用 `1h` 的 `X*/Y*` 作为主假设与参考点集合
- `4h` 计划：主假设优先使用 `4h` 的 `X*/Y*`；若 `4h` 的参考点不足或稳定性不够，允许引入 `1h` 的关键点作为参考点集合的一部分
- `4h` 视为最大周期，不再向更大周期升级

### 8.4.2 触发条件（必须满足）

- 价格已出对应周期的上下轨之外：
  - `1h` 长期计划：要求 `1h` 自身破轨触发
  - `4h` 长期计划：要求 `4h` 自身破轨触发；若用 `1h` 破轨提前触发，只允许生成“预备计划”，仓位更小、过期更快

### 8.4.3 计划假设（thesisRef）与执行参考（entryRefs）

- thesisRef（主假设）：
  - 做多：`Xn (n>1)`
  - 做空：`Yn (n>1)`
- entryRefs（执行参考集合）：
  - 做多：`{X1..X(n-1)}`
  - 做空：`{Y1..Y(n-1)}`

从 entryRefs 构建 entryZone 的简单方法：

- `anchor = median(entryRefs.prices)`（或加权均值：周期权重 4h > 1h）
- `entryZone = [anchor - bufLong, anchor + bufLong]`
- `bufLong` 建议与 `1h/4h bandWidth` 挂钩（比短期更宽）

### 8.4.4 取消/失效条件（长期更严格）

- 若价格在未触达 entryZone 前已经回到对应周期中轨另一侧并维持一段时间，取消计划
- 若生成后出现更深一档预判点（例如从 `X2` 推进到 `X3`）：
  - 可选择升级为新计划（thesisRef=X3，entryRefs=X1..X2），并取消旧计划
  - 或并行存在，但总风险预算不变（必须缩小仓位）

## 9. 仓位与风控（强制规则）

建议把风控当作“策略的一部分”，而不是执行层补丁：

- 单笔风险：`risk = 账户权益 * r`（例如 r=0.2%~0.8%）
- 仓位大小：`qty = risk / |entryMid - SL|`
- 同方向最大同时计划数：短期 1~2、中期 1、长期 1（可配置）
- 触发保护：
  - 同周期同向 breakout 冷却
  - 触发后若价格迅速回到轨道内并穿越中轨，取消计划（避免追尾）

---

## 10. 计划（Plan）模型：分析层输出给执行层的标准化对象

建议每个计划包含：

- `id / symbol / direction / tfTrigger / tfPlan`
- `entryZone: [low, high]`
- `entryStyle`: `maker-limit` / `taker-market`（默认 maker，极端才 taker）
- `sl`: 止损价
- `tp[]`: 目标数组（价格 + 份额）
- `createdAt / expiresAt`
- `state`: `pending / entered / partial / closed / canceled`
- `reason`: 触发原因（破轨周期、breakoutStrength、关键点簇信息、背景 BG）
- `refPoints`: 与关键点编号绑定的字段（建议至少包含）：
  - `entryRef`: 本计划用来构建开仓区间的参考关键点（例如 `X1` 或 `Y1`，以及对应价格/来源周期）
  - `thesisRef`: 本计划的主假设关键点（例如长期计划必须绑定 `X2` 或 `Y2`，允许“未确认”的预判点位）

执行层只认 Plan，不关心策略内部细节。

## 11. 场景示例（按你文档的描述扩展）

### 示例 A：BG_DOWN，下破 5m 下轨

- 背景：4h/1h/30m/15m 都在中轨下方（BG_DOWN）
- 触发：`5m low < 5m dn`，且突破幅度满足阈值
- 关键点：收集 1m/5m 的 X 点聚类
- 若最近 X 簇距离很近：在该簇附近生成短期做多计划，TP1=5m mb，TP2=15m mb
- 若 5m dn 已低于最近 5m X 且 X 点逐步抬高：升级到 15m，使用 15m mb 作为锚生成“更保守”的计划区间

### 示例 B：BG_UP，上破 15m 上轨

对称处理，做空计划优先以 15m/30m 的 Y 簇作为锚，TP1=15m mb，TP2=30m mb 或 1h mb。

## 12. 实现落地建议（最小可行版本 MVP）

如果要最快落地成可跑系统，建议先做 MVP：

1) 只做短期（1m/5m 触发破轨）  
2) 只使用关键点簇 + 触发周期 dn/up 兜底  
3) TP 只做一档：回到触发周期 mb  
4) SL 用触发周期轨道外缓冲  
5) 加冷却 + 单笔风险固定比例

跑起来后再逐步扩展到中期/长期分层目标与更复杂的升级周期逻辑。

## 13. 历史回测方案（方案 A：以 1m 为主时钟，多周期按收盘边界更新）

目标：用“离线实时”的方式复用同一套多周期逻辑（Boll + 震荡结构 X/Y 编号 + Plan/执行分层），在严格避免未来函数的前提下做回测。

### 13.1 主时钟与事件流

- 主时钟：`1m` K 线（按 `open_time` 升序逐根推进）
- 事件源：
  - 每根 `1m` 产生一次 `onBarClose("1m", bar)`
  - 当 `t` 命中更大周期边界时，额外触发：
    - `onBarClose("5m") / onBarClose("15m") / onBarClose("30m") / onBarClose("1h") / onBarClose("4h")`
- 边界判定：
  - 使用 UTC 时间戳与周期秒数对齐（与交易所 bucket 一致），例如 `ts % intervalSec == 0` 视为周期收盘边界

### 13.2 多周期状态更新顺序（同一时刻）

建议固定顺序，保证可复现：

1) 处理 `1m` 收盘（更新 1m Boll/破轨/pending/结构体推进）
2) 若命中更大周期边界，按从小到大依次处理：`5m → 15m → 30m → 1h → 4h`
3) 所有周期状态更新完后，才允许策略生成 Plan（避免同一时刻“用到尚未更新的更大周期状态”）

### 13.3 数据边界与“无未来函数”约束

回测中任何决策在时刻 `t` 只能使用：

- 已收盘的 bar 数据（`open_time <= t` 且该 bar 已完成）
- 已完成的结构体状态（由同样只消费已收盘数据的状态机增量推进得到）

同一根 `1m` 内的成交/止损止盈判定（是否允许 intrabar）需要在实现前确定（见 13.6）。

### 13.4 震荡结构与关键点编号在回测中的产生方式

为了保证回测与实盘语义一致，建议回测过程直接复用 Webman 的震荡结构增量逻辑：

- 每个周期维护一份 `OscillationEngine` 状态（或等价的“pending + inband_count + last_processed_open_time + x_points/y_points”）
- 在该周期的 `onBarClose(interval)` 时，仅喂入这一根已收盘 K 线（增量推进）
- 关键点编号 `X1..Xn / Y1..Yn` 在引擎推进时落入 `oscillation_structures.x_points/y_points`（回测侧读取时直接可用）

### 13.5 回测的三层职责（与实盘一致）

- 数据层（历史数据游标）：
  - 以 `kline_1m` 为主表读取 `[start, end]` 时间窗
  - 需要能按 `symbol` 过滤
- 分析层（PlanEmitter）：
  - 读取多周期状态，按文档信号链生成 Plan（包含 `thesisRef=Xn/Yn(n>1)` 与 `entryRefs={前序点位}`）
- 执行层（SimExecution）：
  - 根据 Plan 生成订单并撮合成交，产出 trades / equity
  - 统一处理手续费、滑点、仓位、保证金、强平（可分阶段实现）

### 13.6 需要在写代码前固定的回测规则（实现必须可复现）

为了让结果可对比、可复现，必须在实现前固定以下规则：

- 成交粒度：
  - 仅 bar-close：信号在 bar 收盘计算，下一根 bar 才允许成交
  - 或 intrabar：用 `1m` 的 `high/low` 判断挂单/止损止盈是否触发
- 同一根 `1m` 同时触发 TP 与 SL 的先后规则（保守建议：先 SL）
- 费用模型：maker/taker 费率（是否区分）、资金费率（是否纳入）
- 滑点模型：固定 bp / 固定价格偏移 / 随波动率变化
- 限价单成交规则：`buy: low <= price`、`sell: high >= price`（是否允许部分成交）

建议先实现“intrabar + 保守先 SL + 固定费率 + 固定滑点”的 MVP，之后再扩展更真实的撮合模型。

### 13.7 回测输出与验收

建议至少输出：

- `trades[]`（开平仓、价格、数量、费用、原因、关联 planId）
- `equityCurve[]`（按时间）
- 汇总指标：胜率、盈亏比、最大回撤、收益回撤比、持仓时间分布、按周期贡献（1m/5m/…）

验收优先级：

1) 同一输入数据多次回测结果完全一致（确定性）
2) 不使用未来数据（可通过“延迟一根成交”与边界日志验证）
3) 与实盘的结构体编号/点位序列在同窗口内一致（抽样对比）

### 13.8 Webman 回测接口草案（实现约定）

- Endpoint：`GET /api/backtest/run`
- 必填参数：
  - `symbol`
  - `start_ms / end_ms`（UTC 毫秒时间戳）
- 回测规则（当前实现）：
  - intrabar：用 `1m` 的 `high/low` 判定挂单成交与 TP/SL 触发
  - 同根 TP/SL：先 SL（保守）
  - 费用：`fee_bps` 固定费率（不区分 maker/taker）
  - 滑点：`slippage_bps` 固定滑点
- 模式参数：
  - `mode=short|long|both`（默认 `short`）
  - `enable_4h_preplan=1|0`：允许 `1h` 破轨触发 `4h` 的“预备计划”（默认 1）
  - `preplan_risk_mult`：预备计划风险倍率（默认 0.3）
  - `preplan_expiry_sec`：预备计划过期秒数（默认 3600）
  - `cooldown_mult`：同方向同计划周期冷却倍数（默认 1.0）
  - `warmup_ms`：回测前热身窗口（用于累积关键点/状态，不计入交易与权益曲线）
